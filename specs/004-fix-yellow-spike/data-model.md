# Data Model: Fix the Bright Yellow LED Artifact in Rainbow-Palette Gradients

This is a rendering bug fix — no persistent data, no wire-format change. The relevant value model:

## Gradient color (`CHSV`)

- Produced by `ColorPalette::GetGradient(fract16 position, bool wrap)` interpolating between palette stops; fields `h`, `s`, `v` (all `uint8_t`).
- Effects typically overwrite `v` after interpolation (brightness policy is per-effect); `h`/`s` carry the palette blend.
- Only *interpolated* colors are flattened; exact stops (`GetColor`) and solid-palette branches are not.

## Total RGB drive (derived)

- `drive(pixel) = r + g + b`, range 0–765. Proxy for LED power draw and perceived pop.
- Baseline: for `hsv2rgb_rainbow` at full s/v, all hue sections render drive 255–256 **except** hue 32–95 (yellow band), which rises to ~341 (Y1 boost). Scales linearly with `v`.
- The fix's post-condition: gradient pixels' drive ≤ reference drive of `CHSV(HUE_RED, s, v)`.

## Reference-vector corpus (`sim/test/vectors/reference.json`)

- 1,380 committed cases: effect × palette × device (scarf/puck/ufo) × timestamps, bytes per LED.
- Regenerated by the `vectorgen` CMake target; consumed by `test/ReferenceVectorTest.cpp` (firmware side) and `sim/test/cases/vectors.test.mjs` (sim side).
- This change intentionally alters bytes for cases of the four affected effects only; `meta.firmwareGitDescribe` also changes. Any other case changing indicates a scoping bug.

## State transitions

None. All rendering is a pure function of `(led_index, time_ms, strip, packet)` — unchanged by this feature.
