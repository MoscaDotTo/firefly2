Candy Kingdom · Firefly2

A field guide to a swarm of lights

Firefly2 is the firmware and hardware behind wireless-synchronized LED devices — bikes, costumes, lanterns, art pieces — that find each other over 915 MHz radio, elect a leader, agree on a clock, and animate in lockstep with no infrastructure at all.

01 · Overview

What the system is

Every Firefly device is a small battery-powered board driving one or more WS2812B (NeoPixel) strips, with an RFM69HW packet radio on 915 MHz. Devices form an ad-hoc mesh: there is no pairing, no join procedure, and no configuration beyond compiling in a DeviceDescription. Power a device on and it listens; if it hears nobody for a few seconds, it appoints itself master and starts running the show.

The master does three things on timers: it broadcasts a heartbeat carrying the network clock every second, re-broadcasts the current effect selection every two seconds, and picks a new random effect and palette every sixty seconds. Every other device is a slave: it locks its clock to the heartbeats and renders whatever effect the master last announced. Because all animation is computed as a pure function of (LED index, network time, strip config, effect packet), devices that agree on those inputs render identical frames without ever exchanging pixel data.

Two auxiliary device roles exist today: a handheld controller with buttons that lets a human force an effect, palette, or solid color across the swarm, and a DMX bridge on ESP32 that translates a DMX universe into direct color commands — the precedent for the planned 2.4 GHz bridge.

03 · Radio protocol

Packets & wire format

The application layer defines exactly four packet types (lib/radio/Radio.hpp). A Firefly packet is a 3-byte header plus up to 58 bytes of type-specific payload, carried inside RadioHead’s own RF69 framing (preamble, sync words, length byte, 4-byte RadioHead header, CRC — the RFM69 FIFO’s 64 bytes bound the total).

Radio configuration lives in RadioHeadRadio::Begin(): 915.0 MHz, +13 dBm, and otherwise RadioHead RH_RF69 defaults — which means the library’s default modem settings (GFSK 250 kb/s) and default 2-byte sync word shared by every RadioHead project in the world. There is no addressing, no encryption, and no network identifier at any layer.

04 · Radio protocol

Mesh flooding

NetworkManager implements the entire mesh in about thirty lines: it is a flood network with duplicate suppression, not a routed mesh. Every node re-broadcasts every packet it hasn’t seen recently, extending range one hop at a time.

  • Send: stamp the packet with a fresh random packet_id in [2, 0xFFFF), transmit, and remember the ID.
  • Receive: if the ID is in the recent-IDs cache, drop it silently. Otherwise immediately re-transmit the packet verbatim, cache the ID, and hand it up to the state machine.
  • Cache: a 5-entry circular buffer (kRecentIdsCacheSize). At the master’s steady-state rate (~1.5 packets/s) that is ~3 seconds of memory.
Airtime economics. Every packet is transmitted once per node that hears it. A swarm of N mutually-audible devices turns each logical broadcast into N transmissions. At the steady-state rate of ~1.5 packets/s this is fine for dozens of nodes on 915 MHz, but it is the single most important number when reasoning about coexisting swarms — two groups in earshot don’t just share spectrum, they re-broadcast each other’s packets.

05 · Radio protocol

Master / slave lifecycle

RadioStateMachine (src/generic/) is a two-state machine processing one event per tick — either a received packet or an expired timer, never both. That constraint keeps the handlers simple and is load-bearing for the tests.

States and transitions

Slave

  • Syncs clock from heartbeats
  • Applies SET_EFFECT / SET_CONTROL
  • Rebroadcasts everything
  • Arms a 5–7 s dead-man timer
no packet for 5000 + rand(2000) ms
lost election, or CLAIM_MASTER received

Master

  • Heartbeat every 1 s
  • Rebroadcast effect every 2 s
  • Random new effect every 60 s
  • Elects on rival heartbeat

Election

When a master hears a rival heartbeat, it draws a random uint16 and compares it to the rival packet’s packet_id. Higher number wins: the winner broadcasts CLAIM_MASTER, the loser demotes itself. Ties resolve over successive heartbeats — with randomized slave timeouts and a coin-flip election, the network converges to one master within a few seconds. There is no seniority, and mastership is deliberately cheap to lose: any device can be master, so the swarm survives any device leaving.

Clock & cadence continuity

Network time is the master’s millis() propagated by heartbeat; each slave stores a signed offset. Two details preserve visual continuity across master handoffs: a new master inherits network time (it sends GetNetworkMillis(), not its raw clock), and it inherits the effect-change cadence via effect_change_seen_at_ — so a leadership change doesn’t make the lights jump or reset the 60-second rotation.

06 · Light

LED pipeline

LedManager owns the effect registry and the per-frame render loop; FastLedManager binds it to real hardware through FastLED. Each frame, RunEffect() walks every strip in the DeviceDescription and asks the current effect for each LED’s color at the current network time — local time never touches rendering.

  • Strip flags shape the query: Reversed flips the index, Off blanks the strip, Dim divides output by 8, and Circular/Mirrored/Tiny/Bright/Controller are hints effects use to adapt (e.g. wrap seamlessly on a ring).
  • Power safety: FastLED.setMaxPowerInVoltsAndMilliamps(5, milliamps_supported) caps total draw per the device’s power budget (2350 mA on the stock RF board).
  • LED zero is the onboard pass-through pixel; user strips start at index 1, so SetLed(i) writes leds[i+1].
  • Effect index is network state: SET_EFFECT carries a raw index into the weighted registry, so every device must register the same effects in the same order — the registry in LedManager’s constructor is effectively part of the protocol.

07 · Light

Effects

Effects are stateless-by-convention subclasses of Effect implementing one method: GetRGB(led_index, time_ms, strip, setEffectPacket). 09 · Configuration

Device catalog

A DeviceDescription is a power budget plus a list of StripDescriptions (LED count + flags). The fleet is declared in lib/device/Devices.hpp; Devices::current points at one entry — currently scarf — which serves as the compiled-in fallback. Node and fancy-node builds default to READ_FROM_FLASH: at boot they parse a 32-byte FlashConfigV1 blob (magic "FFLY", version, strip table, CRC32) from the config flash region (0x1E000 on node), falling back to Devices::current when absent or invalid. Because the format is a serialized POD rather than a raw struct, external tools — notably the web configurator — can write it directly, so reconfiguring a device doesn’t require recompiling. WRITE_TO_FLASH seeds the blob from the header (write-skipped when unchanged to avoid flash wear).

11 · Hardware

Firmware entry points

Every device follows the same skeleton; the differences are peripherals and boot rituals.

// The canonical device loop (node.cpp, simplified)
RadioHeadRadio *radio = new RadioHeadRadio();  // must be heap-allocated —
NetworkManager nm(radio);                        // stack object crashes on RX
RadioStateMachine state_machine(&nm);
FastLedManager *led_manager;                     // built from Devices::current or flash

void loop() {
  state_machine.Tick();      // radio, election, timers (runs twice internally)
  led_manager->RunEffect();  // render frame at network time
  // node: feed 128 ms watchdog every 5th iteration
  // fancy-node/controller: battery filters, low-power cutoff
}

Boot rituals worth knowing: the node configures the SAMD21 watchdog against the ultra-low-power 32 kHz oscillator (no prescaler — feeding is expensive); the controller sets the nBOOT_SEL option bit on first boot and uses RTC epoch sentinels to detect a quick double reset and jump to the STM32 system bootloader; radio init failure is signaled by a red blink loop (FatalErrorAnimation).

14 · Looking ahead

Known sharp edges

Documented honestly, because several of these become design inputs for the next revision rather than bugs to ignore.

  • No network identity or authentication. Any RFM69 running RadioHead defaults on 915 MHz is a full peer: it can claim mastership, set effects, and its packets get rebroadcast by your swarm. Two Firefly groups camped near each other will merge into one show (and burn each other’s airtime doing it). The RFM69 has hardware AES-128 and configurable sync words — both unused today.
  • Dedup is probabilistic. Packet IDs are random 16-bit values checked against a 5-entry cache; an ID collision inside the window silently drops a legitimate packet. Fine at 1.5 pkt/s, worth revisiting at any higher rate or with multiple bridged segments.
  • The DMX bridge sidesteps the mesh. It stamps sequential IDs starting at 0 and never rebroadcasts, so its packets are deduped by others but it contributes nothing to propagation — acceptable for a stationary transmitter, but not a pattern to copy for the 2.4 GHz bridge.
  • The effect registry is implicit protocol. SET_EFFECT ships a raw index, so all devices must be built with identical registries. Mixed-firmware swarms render different effects for the same packet (indices wrap on hardware rather than asserting).
  • Timer comparisons don’t handle millis() rollover (~49.7 days) — irrelevant at festival timescales, relevant if “camp structure” devices stay powered for weeks.
  • Folk-knowledge bugs are load-bearing: RadioHeadRadio must be heap-allocated or the node crashes on receive; the first Tick() may only run the radio once (suspected compiler bug, see comment in RadioStateMachine::Tick).

15 · Looking ahead

Constraints on the 2.4 GHz bridge

The next revision plans an ESP32-based, 2.4 GHz-only tier for stationary objects (camp structures), bridged to the existing 915 MHz swarm — with an eye toward productization and multiple co-located swarms. The current protocol dictates most of the bridge’s shape:

What the bridge must preserve

  • One dedup domain. The mesh’s only loop-prevention is the packet-ID cache. A bridge relaying packets between radios must carry packet_id across unchanged and keep its own (larger) seen-cache, or a two-bridge deployment creates a packet loop that floods both bands.
  • Clock semantics. Slaves sync to heartbeat time minus propagation ≈ 0. Any bridge buffering adds skew directly into animation phase across the seam; relay heartbeats with minimal, ideally constant, latency.
  • The effect registry contract. 2.4 GHz devices interpret the same SET_EFFECT indices, so they inherit the lockstep-registry requirement — a versioning story is needed before shipping firmware to strangers.
  • Election containment. If bridged naively, a 2.4 GHz device can win mastership of the combined network. Decide explicitly whether the bridge forwards CLAIM_MASTER/heartbeats bidirectionally (one merged swarm) or acts as the 2.4 GHz segment’s permanent master proxy (simpler, and the DMX bridge already demonstrates the transmit-only posture).

Swarm identity — the cheapest place to add it is 915 MHz, not just 2.4

The competing-swarms problem already exists on the current band. Two mechanisms are available on the RFM69 without protocol redesign, and both double as the “network ID” you’re considering:

  • Sync words as network ID. RadioHead’s setSyncWords() makes the radio hardware drop other networks’ frames before they cost the MCU anything — the classic RFM69 network-ID mechanism. Zero airtime cost, but zero security: it’s an identifier, not authentication.
  • Hardware AES-128 as membership. RH_RF69::setEncryptionKey() makes possession of the key the definition of swarm membership — wrong-key frames decrypt to garbage and fail parsing. Cost: payload pads to 16-byte blocks (Firefly packets are ≤7 bytes of payload today, so airtime roughly doubles per frame — at 1.5 pkt/s this is affordable). Note this authenticates membership, not messages: no replay protection, and anyone with the key is fully trusted.

Neither stops a hostile actor from jamming, but both solve the actual stated problem — accidental cross-swarm flooding and rebroadcast amplification between neighboring groups. A reasonable v3 posture: sync-word network ID for isolation, AES key per swarm for product deployments, same key/ID provisioned to the bridge’s 2.4 GHz side (ESP-NOW’s per-peer encryption or an app-layer MAC — note ESP-NOW broadcast frames are unencrypted, so app-layer is likely required for broadcast-style traffic).

Airtime on 2.4 GHz is a different economy

915 MHz at festivals is relatively quiet; 2.4 GHz is the most congested spectrum on site (camp Wi-Fi, phones, BLE). The flood-rebroadcast design that is cheap on 915 should not be transplanted as-is: prefer a protocol with native broadcast and no per-node re-transmission (ESP-NOW broadcast, or ESP32 Wi-Fi in a fixed-infrastructure mode for stationary structures), and let only designated repeaters extend range. The current protocol’s tiny packets and 1.5 pkt/s steady state are an asset here — the payload fits any 2.4 GHz transport trivially.
